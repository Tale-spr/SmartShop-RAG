# 数据收集方案

## 目标
为“面向中小型小家电品牌商家的电商客服混合检索 RAG 系统原型”确定一个适合第一阶段落地的细分品类，并给出一套可执行的数据收集方案。当前目标不是追求大而全，而是先做出一套 **来源可信、场景统一、可清洗、可解释** 的知识库。

## 品类选择原则
选择第一阶段品类时，优先考虑以下标准：

1. 客服问题是否足够高频且真实。
2. 知识是否天然分散在商品详情、参数页、FAQ、售后说明中。
3. 商品问法是否同时包含“语义问题”和“精确词问题”，从而有混合检索的发挥空间。
4. 公开资料是否容易获取，且便于后续说明数据来源。
5. 是否能避免与旧项目场景过度重叠，形成新的项目识别度。

## 推荐品类备选

### 方案 A：空气炸锅 / 多功能锅

#### 推荐度
**最高，优先推荐作为第一阶段落地品类。**

#### 原因
- 用户问题天然丰富，覆盖售前与售后两类典型客服问题。
- 商品知识分散明显，常见于商品详情、参数页、菜谱说明、清洗方式、售后规则等多个页面。
- 问题既有语义类问法，也有精确参数类问法，适合混合检索。
- 数据相对容易获得，公开商品说明、FAQ 和使用指南较多。

#### 典型用户问题
- 这款适合几个人用？
- 能不能烤鸡翅/蛋挞？
- 容量 5L 和 6L 差别大吗？
- 清洗麻不麻烦？
- 内胆能不能进洗碗机？
- 售后保修多久？

#### 评价
这是一个非常适合做第一版 RAG 项目的品类。它既不会过于简单，也不会像数码/金融那样对术语和专业知识要求过高。对面试官来说也很好理解。

---

### 方案 B：电饭煲 / 电压力锅

#### 推荐度
**高，作为第二推荐。**

#### 原因
- 商品参数明确，容量、内胆材质、加热方式、预约功能等都适合做知识问答。
- 售前差异对比需求强，用户常问“3L 和 5L 有什么区别”“IH 和底盘加热差别是什么”。
- 售后也有清洗、保修、故障现象等稳定问法。

#### 典型用户问题
- 这款适合 2 个人还是 4 个人用？
- 预约煮饭怎么设置？
- 内胆是什么材质？
- 压力锅排气正常吗？
- 保温多久会影响口感？

#### 评价
这个品类的优势是参数体系很清晰，适合做“参数词 + 语义问法”的混合检索验证。缺点是问题类型相对比空气炸锅略窄，内容表达也稍微更规整一些。

---

### 方案 C：加湿器 / 电风扇 / 环境电器

#### 推荐度
**中等，可作为备选。**

#### 原因
- 用户问题具有明显场景性，比如噪音、风力、加湿量、是否适合卧室或婴儿房。
- 问法更口语化，更能体现语义检索价值。
- 商品说明中也会涉及参数、面积、模式、噪音等级等精确字段。

#### 典型用户问题
- 放卧室会不会太吵？
- 一晚上要加几次水？
- 适合多大房间？
- 有宝宝能不能用？
- 怎么清洗水箱？

#### 评价
这个方向的问题很贴近真实生活场景，语义表达丰富，适合展示 query rewrite 与混合检索。但数据来源可能比厨房类小家电略分散，商品说明质量也不一定稳定。

---

### 方案 D：吸尘器 / 洗地机

#### 推荐度
**不建议作为第一阶段首选。**

#### 原因
- 问题确实丰富，客服场景也很典型。
- 但它和你之前扫地机器人项目的语义空间过近，面试时容易被认为只是换壳延续。
- 你现在需要的是一个“更干净的新项目定位”，而不是让新项目继续贴着旧项目的家居清洁场景。

#### 评价
如果后面想扩品类或做第二阶段数据，可以再考虑。但不适合作为当前重构项目的首发品类。

## 最终推荐顺序
1. **空气炸锅 / 多功能锅**
2. **电饭煲 / 电压力锅**
3. **加湿器 / 电风扇 / 环境电器**
4. **吸尘器 / 洗地机**

## 当前建议
如果没有特殊限制，第一阶段建议直接选择：

**空气炸锅 / 多功能锅**

理由是：
- 用户问题丰富且真实
- 兼具语义问法和参数问法
- 知识来源公开、好找、好清洗
- 与旧项目区分度足够
- 最适合拿来验证“向量检索 + BM25 + 重排”是否确实带来提升

## 数据收集目标
第一阶段数据收集目标不是做一个“大型电商知识库”，而是做一个 **小而硬的垂直知识库**。

建议范围：
- 选择 **1 到 2 个品牌**
- 每个品牌选择 **3 到 6 个商品**
- 总体整理出 **30 到 80 篇文档级知识内容**

这样做的好处：
- 场景统一
- 便于说明来源
- 便于后续做检索对照实验
- 不会把精力耗在低质量大规模数据清洗上

## 数据来源优先级

### 第一优先级：官方商品说明与售后资料
优先收集：
- 商品详情页
- 商品参数页
- FAQ 页面
- 使用说明 / 清洁说明
- 售后政策 / 保修说明 / 退换货说明

#### 原因
- 来源可信
- 信息结构相对清晰
- 更适合作为知识库主干
- 面试时更容易说明“数据从哪里来、为什么可信”

### 第二优先级：公开商品问答区
优先收集：
- 用户问商家答
- 商品详情下的常见问题区

#### 原因
- 更接近真实客服问法
- 能补足自然语言问题样本
- 有助于后续构造检索测试 query

### 第三优先级：公开商品评论中可提炼的问题
只作为后续补充，不建议第一阶段就大量引入。

#### 原因
- 噪声大
- 情绪表达多
- 评论不能直接等同于知识答案
- 清洗成本高

## 推荐数据结构
建议在 `data/knowledge_base/` 下按品类、品牌、商品组织。

### 推荐目录结构
```text
data/knowledge_base/
  air_fryer/
    brand_a/
      product_1/
        detail.md
        specs.md
        faq.md
        usage.md
        aftersales.md
      product_2/
        ...
    brand_b/
      ...
```

### 每类文档建议内容
- `detail.md`：商品卖点与适用场景
- `specs.md`：容量、功率、尺寸、模式、材质等参数
- `faq.md`：常见问答
- `usage.md`：使用方式、清洗说明、注意事项
- `aftersales.md`：保修、退换货、售后规则

## 数据收集步骤

### 第一步：先定品牌和商品
建议你先从一个品类里选：
- 1 到 2 个品牌
- 每个品牌 3 到 6 个商品

目标是让知识库既有横向差异，也不会规模失控。

### 第二步：先采“知识主干”
先收：
- 详情页
- 参数页
- FAQ
- 使用说明
- 售后说明

先不要急着抓评论。

### 第三步：整理成文档级知识文件
不要一开始就做复杂结构化。先把文档整理成：
- `.md`
- `.txt`

保证：
- 每份文档主题单一
- 来源清楚
- 不混多个商品

### 第四步：单独整理一份“问题样本集”
建议额外维护一个文件，用于记录真实问法，比如：
- 用户自然语言问法
- 对应商品/场景
- 预期应命中的文档类型

这份问题样本集后续可用于：
- query rewrite 测试
- 检索对照实验
- 回答质量人工检查

## 第一阶段不建议做的事
- 不要一开始就抓全平台数据。
- 不要一开始就把评论当主知识源。
- 不要一开始就跨多个大品类。
- 不要先追求海量数据，而忽视来源可信度和文档边界。

## 当前建议的执行路径
### 推荐品类
**空气炸锅 / 多功能锅**

### 推荐知识源组合
1. 官方商品详情页
2. 官方参数页
3. FAQ / 常见问题
4. 使用说明 / 清洁说明
5. 售后 / 保修 / 退换货说明
6. 少量公开问答区内容（作为自然语言问题补充）

### 推荐第一阶段成果
- 一个统一品类的小型知识库
- 一批可解释来源的客服问题样本
- 后续可直接用于验证：
  - 单向量检索
  - BM25
  - 混合检索
  - rerank
