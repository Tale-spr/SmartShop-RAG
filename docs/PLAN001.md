# 型号未确认场景修正计划

## Summary
修正当前系统把“检索命中的某个型号”误当成“用户真实持有型号”的问题。目标是让系统在型号未确认时优先澄清，不再直接下结论“您买的是 MF-XXX”，同时保留现有混合检索链路和对外接口不变。

## Key Changes
### 1. 在检索链路中新增“型号确认状态”
- 在 `rag_service` 的 `retrieve_docs()` 结果中增加结构化判定字段，至少包含：
  - `detected_query_models`：从原始 query 和 rewrite 后 query 中识别到的型号词
  - `retrieved_models`：最终入选文档涉及到的型号集合
  - `model_confirmation_status`：固定为 `confirmed / unconfirmed / ambiguous`
- 判定规则固定：
  - 用户 query 中明确出现某个型号词，且最终证据与该型号一致：`confirmed`
  - 用户 query 中没有明确型号词：`unconfirmed`
  - 用户 query 中有型号词，但最终证据涉及多个型号或与 query 中型号不一致：`ambiguous`
- `retrieval_trace` 记录以上字段，便于前端和后续实验排查。

### 2. 在最终回答阶段增加“型号未确认”护栏
- `react_agent` 在生成最终回答前读取 `retrieval_trace` 中最近一次的 `model_confirmation_status`。
- 当状态为 `unconfirmed` 或 `ambiguous` 时，回答阶段必须遵守：
  - 不得把任何检索到的型号直接表述成“您购买的就是这款”
  - 不得输出“您这款/您买的这款是 MF-XXX”这种确定句
  - 允许使用“如果您说的是 MF-XXX”“目前检索到的某个候选型号是 MF-XXX”这类非确认表达
  - 优先建议用户查看机身铭牌、说明书封面、订单页或提供外观特征
- 当状态为 `confirmed` 时，才允许基于该型号给出具体容量、功率、功能差异说明。

### 3. 强化主回答 prompt 的约束
- 在 `main_prompt.txt` 中补充硬规则：
  - 如果用户没有明确提供型号，且检索证据无法唯一确认型号，不得把某个检索命中的型号当作用户真实商品
  - 此时应先说明无法确认具体型号，再给出核验方式
  - 如果要引用候选型号，只能以“候选型号/可能型号”方式表达
- 保留现有“资料不足要明确说明”“不要编造商品信息”的规则，不改整体回答风格。

### 4. 增加轻量型号识别与候选分布逻辑
- 在检索层增加一个轻量的型号词识别函数，优先通过正则识别类似：
  - `MF-KZ...`
  - 容量/功率等型号伴随词
- 不依赖 LLM 做型号确认，避免把“型号确认”本身变成一个高波动步骤。
- 对最终 top docs 统计型号分布：
  - 单一型号占据全部最终证据，只能说明“候选集中到某型号”
  - 只有 query 明示或后续显式确认，才能升级为用户真实型号
- `shared/policies` 文档不参与型号确认统计。

### 5. 对话体验修正策略
- 对于“我不知道我买的这个锅容量是多少”“我这款是几升”这类 query：
  - 默认先进入“型号待确认”答法
  - 可以补一个简短帮助路径：
    - 看机身铭牌
    - 看说明书封面
    - 看订单页
    - 提供颜色、旋钮/面板样式、是否有可视窗等外观信息
- 对于“真的嘛？我买的好像不是 MF-XXX”这类用户纠错：
  - 系统应承认上一轮型号判断不成立
  - 明确说明“前一轮只是基于检索到的候选资料，不代表已确认您的型号”

## Test Plan
- 新增或修改测试覆盖以下场景：
  - `query` 不含型号、top docs 来自单一型号：回答中不能出现“您买的就是 MF-XXX”
  - `query` 不含型号但问容量：回答应先澄清型号未确认，并给出核验路径
  - `query` 含明确型号且检索一致：可以直接回答具体容量/参数
  - `query` 含明确型号但检索结果混入其他型号：回答应避免直接确认，改为提示信息可能不一致
  - 用户第二轮明确否认系统猜测型号：回答应承认未确认并回到澄清流程
- 验收标准：
  - 当前截图中的错误案例不再出现
  - `retrieval_trace` 中能看到 `model_confirmation_status` 和型号分布
  - 不影响现有 `/chat`、Streamlit 和混合检索主链路
  - 对已明确型号的问题，回答能力不下降

## Assumptions
- 这版修正只处理“型号确认”问题，不扩展到颜色、容量、功率等所有属性确认逻辑。
- 第一版采用规则判定，不增加新的模型调用。
- 不修改 API schema，只在内部 `retrieval_trace` 和回答约束上增强。
- 默认以“宁可先澄清，也不误认型号”为优先策略。
