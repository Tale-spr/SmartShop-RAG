# 项目重构方案

## 重构结论

当前 `智扫通 Agent` 项目建议从“多能力混合型 Agent Demo”收缩为一个**面向电商客服场景的混合检索 RAG 项目**。

新的项目目标不再是同时覆盖：

- Agent 工作流
- 报告生成
- 长期记忆
- 自定义评测

而是聚焦在一条主线：

**混合检索 RAG：向量检索 + BM25 + rerank + grounded answer generation**

---

## 仓库策略建议

### 推荐方案

**推荐新建一个 GitHub 仓库，并把当前项目归档为旧版本。**

更具体地说：

1. 复制当前文件夹作为新项目起点  
2. 在 GitHub 新建一个新的公开仓库，用来承载“电商客服混合检索 RAG 项目”  
3. 当前这个 `智扫通 Agent` 项目仓库改成 `private` 或保留为归档仓库  

### 为什么不建议直接在当前仓库上改

如果直接在当前仓库大改，会有几个问题：

- 旧项目的历史提交、文档和命名会持续混入新项目语境
- 当前仓库里已经有大量 Agent / report / evaluation 相关实现，后续删改成本高
- 面试官或 GitHub 访问者会更难一眼看懂新项目的主线到底是什么
- 你后面还要单独做“记忆项目”，如果第一个项目定位已经不够纯，两个项目之间边界会变得模糊

### 为什么新建仓库更适合现在的目标

新仓库的优势是：

- 项目定位干净，直接就是“RAG 项目”
- README、目录结构、接口设计可以从第一天就围绕 RAG 写
- 不需要额外解释“为什么这里还有 report / evaluation / memory 的旧痕迹”
- 更利于后续和第二个“记忆项目”形成清晰分工

### 最终建议

结论上，我建议：

**复制当前文件夹，新建 GitHub 仓库，把原来的 Agent 项目设为 private 或归档，不在原仓库上继续大改。**

---

## 新项目定位

### 场景

建议切换到：

**电商客服场景**

优先选择电商场景而不是继续扫地机器人场景，原因是：

- 电商客服的数据集和公开资料更容易获取
- 商品问答、售后规则、物流说明、退换货政策等都天然适合做 RAG
- 电商场景更容易解释混合检索、rerank、citation 和 grounding 的价值
- 面试官更容易理解，不需要额外建立业务背景

### 项目一句话定义

**一个面向电商客服场景的混合检索 RAG 系统，支持 query rewrite、向量检索、BM25 检索、结果融合、rerank 和基于证据的回答生成。**

---

## 新项目主线

### 保留的能力

建议保留：

- `FastAPI`
- `Streamlit`
- 知识库构建 / ingest
- 向量库
- 模型工厂
- prompt 管理
- 基础日志
- 检索链路可观测性

### 删除或不再延续的能力

建议移除或不再延续：

- 报告链路
- `report_prompt`
- 报告记忆
- `records.csv` 驱动的月报 / 趋势分析
- 自定义三层 evaluation
- 面向 Agent 的显式工作流叙事
- 长期记忆相关逻辑
- 复杂工具编排

### 新主流程

新项目建议只保留一条核心链路：

1. 用户问题输入
2. query rewrite / query normalize
3. 混合检索
   - 向量检索
   - BM25 检索
4. 结果融合
5. rerank
6. 构建证据上下文
7. 生成最终回答
8. 展示引用证据 / 检索来源

---

## 技术改造建议

### 1. 检索层重构

新项目建议重点实现：

- 向量检索
- BM25 检索
- 混合召回
- rerank

优先目标：

- 不再只依赖单一路径向量检索
- 支持短 query、关键词 query 和长 query 的不同召回特征
- 保证最终回答尽可能 grounded

### 2. 可观测性保留，但只保留“检索链路可观测性”

建议保留：

- 原始 query
- rewrite 后 query
- 向量检索 top-k
- BM25 top-k
- merge 后候选
- rerank 后 top-k
- 最终被引用的证据片段

建议放弃：

- Agent trace
- stop_reason
- tool sequence
- report sequence

原因是：

新项目不再主打 Agent workflow，而主打 RAG 检索质量，所以可观测性应该围绕 retrieval 而不是 tool orchestration。

### 3. 数据集要求

新项目的数据源必须避免继续使用 demo 级合成数据。

建议使用：

- 电商商品详情数据
- 商品 FAQ
- 用户评价
- 售后规则
- 物流规则
- 店铺客服问答公开数据

优先选择：

- 开源、公开、可复现的数据源
- 有明确出处的数据
- 能支撑 retrieval / rerank / answer grounding 的文本型数据

### 4. 评测策略调整

建议不再保留当前自定义三层 evaluation 模块作为新项目主卖点。

可以改成：

- 检索实验对比
  - vector only
  - BM25 only
  - hybrid retrieval
  - rerank 前后对比
- 案例级人工分析
- 必要时接入更通用、社区更认可的 RAG 评估方式

新的评测口径重点应放在：

- 检索命中率
- 召回质量
- rerank 提升效果
- grounded answer 质量

而不是继续强调 Agent workflow 的 rule-based 评测。

## 分阶段实施建议

### 第一阶段：仓库拆分与目标收缩

目标：

- 复制当前项目
- 新建 GitHub 仓库
- 删除 report / evaluation / memory 相关内容
- 把项目名称、README、目录结构全部改成 RAG 项目口径

验收标准：

- 项目对外描述中不再出现 Agent report / evaluation 主线
- README 一眼可见“电商客服混合检索 RAG”

### 第二阶段：知识库与数据集重建

目标：

- 替换原扫地机器人知识库
- 引入可信电商客服数据
- 重新设计文档切分和 ingest 逻辑

验收标准：

- 知识库内容完全切到电商客服场景
- 数据来源可解释、可复现

### 第三阶段：混合检索与 rerank 实现

目标：

- 增加 BM25 检索
- 实现 hybrid retrieval
- 加入 rerank
- 记录 retrieval trace

验收标准：

- 支持 vector / BM25 / hybrid 三种模式
- 能展示 rerank 前后效果差异

### 第四阶段：回答生成与证据展示

目标：

- 构建 grounded answer generation
- 引用检索到的证据片段
- 保证回答风格自然、非模板化

验收标准：

- 回答能清楚基于检索结果生成
- 前端或接口可返回证据来源摘要

### 第五阶段：项目包装

目标：

- 完善 README
- 增加系统架构图
- 补充 Demo 截图
- 准备面试话术

验收标准：

- 仓库具备独立展示能力
- 面试时可以明确讲清项目定位、改进点和技术选择

---

## 与第二个“记忆项目”的边界

为了避免两个项目互相重叠，建议从一开始就明确边界：

### 当前项目

主打：

- RAG
- 混合检索
- rerank
- grounding

不主打：

- 长期记忆
- 用户偏好召回
- 多轮个性化上下文管理

### 第二个项目

主打：

- 会话记忆
- 长期记忆
- 记忆更新
- 记忆召回
- 个性化对话

不主打：

- 多路检索
- 大规模知识库优化

这样两个项目在简历和面试中就会非常清楚：

- 一个证明你会做检索系统
- 一个证明你会做记忆系统

---

## 最终建议

这次重构不应该被理解为“继续修补一个 Agent 项目”，而应该被理解为：

**把当前项目收缩并重塑成一个更纯、更可信、更适合面试展示的混合检索 RAG 项目。**

如果只保留一句话作为执行原则，可以记成：

**砍掉不够硬的模块，保留真正能打的主线；从“什么都做一点”改成“把 RAG 做深”。**
