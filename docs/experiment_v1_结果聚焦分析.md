# experiment_v1 结果聚焦分析

## 观察到的主要现象

### 1. `hybrid_rerank` 值得保留
从当前小样本知识库的结果看，`hybrid_rerank` 相比单向量检索有明确价值，尤其体现在以下场景：

- 容量词或特征词明确、但 query 本身没写完整型号时，`hybrid_rerank` 更容易把目标型号排到前面。
- 单型号问答中，`hybrid_rerank` 对最终 top docs 的排序通常比 `vector` 更稳。
- 一部分多型号对比问题中，`hybrid_rerank` 能比 `vector` 更好地把正确的对比对象同时召回。

典型正例：

- `af_001`
  - query: `3L 这款一个人用够吗？`
  - `vector` 前两条 docs 分别是 `MF-KZE7001`、`MF-KZE5004`，正确目标 `MF-KZ30E201` 只排到第 3。
  - `hybrid_rerank` 能把 `MF-KZ30E201` 提到首位，回答明显更稳定。
- `af_021`
  - query: `4.5L 这款更适合两个人还是三个人用？`
  - `vector` 和 `hybrid` 没能稳定命中 `MF-KZE459X9BD`，回答倾向于“资料不足”。
  - `bm25` 和 `hybrid_rerank` 能命中 `MF-KZE459X9BD`，`hybrid_rerank` 的最终回答更自然。
- `af_043`
  - query: `3L 款和 5L 款主要差别是什么？`
  - `hybrid_rerank` 能同时把 `MF-KZ30E201` 与 `MF-KZE5004` 排到前面，明显优于 `vector` 的局部命中。

结论：当前没有理由回退到纯 `vector`。第一版 hybrid retrieval 是成立的。

### 2. “型号未确认却被系统默认成某型号”是系统性问题
这是当前最明显的可信度问题，不是单个 case 的偶发错误。

在 23 条“query 文本本身未明确给出型号”的问题里，四种模式中回答里直接出现具体型号的次数分别为：

- `vector`: 14
- `bm25`: 14
- `hybrid`: 15
- `hybrid_rerank`: 14

这说明问题不是某个检索模式单独造成的，而是当前系统的整体行为问题：

- 检索链路把某个型号排到前面；
- 生成阶段把“候选型号”误说成了“用户真实购买型号”。

典型反例：

- `af_001`
  - query: `3L 这款一个人用够吗？`
  - 四种模式都会直接提到 `MF-KZ30E201`，但用户并没有明确给出型号。
- `af_011`
  - query: `5L 这个型号一般适合几个人吃？`
  - 四种模式都会直接落实到 `MF-KZE5004`。
- `af_019`
  - query: `带可视窗那款有什么实际用处？`
  - 回答里会直接落实某个候选型号，而不是先承认“当前还不能确认您买的是哪一款”。

结论：这已经足够判定为系统性问题，且优先级高。

### 3. 知识外问题当前主要是“保守兜底”，不是“强编造”
边界外问题目前没有表现出大面积的强幻觉，整体上还是倾向于保守回答“资料不足”或“不支持确认”。

典型样本：

- `af_053`
  - query: `空气炸锅烤红薯一般多少度多少分钟？`
  - 四种模式基本都给出“资料不足”，没有强行编时间温度。
- `af_055`
  - query: `7001 的滤油盘如果单独坏了，配件多少钱？`
  - 四种模式均未强行给价格。
- `af_057`
  - query: `除了空气炸锅，你们有没有同系列微波炉适合宿舍？`
  - 四种模式都没有把空气炸锅知识硬扩展到微波炉。
- `af_058`
  - query: `我想买个能联网远程控制的空气炸锅，你这几款支持吗？`
  - 回答基本都保持保守，没有明显补脑成功能支持。

结论：知识外问题当前不是最优先的阻塞项。

### 4. `hybrid_rerank` 在部分“全局比较类问题”上存在回归
虽然 `hybrid_rerank` 整体值得保留，但它并不是当前所有场景下都更优。

典型反例：

- `af_056`
  - query: `这几款空气炸锅哪一款最省电？`
  - `vector` / `hybrid` 至少能把 `MF-KZ30E201` 的 `1325W` 和 `MF-KZC6054` 的 `2000W` 放进候选范围。
  - `hybrid_rerank` 的最终 docs 丢掉了 `MF-KZ30E201`，转而使用 `MF-KZE7001` 与 `MF-KZC6054` 做比较，最后得出 `MF-KZE7001` 最省电，这明显不稳。

这说明当前 rerank 对“哪一款最…… / 全局比较型 query”的候选保留策略还不够好，容易把真正关键的低功率型号排掉。

结论：这是第二优先级问题，但优先级仍低于“型号未确认护栏”。

## 典型正例

- `af_021`: `hybrid_rerank` 和 `bm25` 能命中 `MF-KZE459X9BD`，优于 `vector`。

- `af_001`: query 未确认型号，但回答直接落实为 `MF-KZ30E201`。
- `af_011`: query 未确认型号，但回答直接落实为 `MF-KZE5004`。
- `af_019`: query 只说“带可视窗那款”，回答却默认成具体型号。
- `af_056`: `hybrid_rerank` 在“最省电”这种全局比较问题上把关键型号排掉，出现回归。

## 当前最优先修复项

### 优先级 1：补“型号未确认护栏”
当前最先要修的不是 rerank，也不是知识外兜底，而是“型号未确认却直接确认型号”的生成护栏。

原因：

- 这是高频、可复现、系统性问题；
- 它直接破坏回答可信度；
- 即使检索命中了相对正确的候选型号，也不代表用户真实买的就是那一款。

建议下一步实现：

- 在 retrieval trace 中加入：
  - `detected_query_models`
  - `retrieved_models`
  - `model_confirmation_status`
- 当 `model_confirmation_status` 为 `unconfirmed` 或 `ambiguous` 时：
  - 禁止回答“您这款就是 MF-XXX”；
  - 改为提示用户查看机身铭牌、说明书封面、订单页，或提供外观特征辅助判断。

### 优先级 2：修全局比较类 query 的 rerank 策略
在型号护栏修完后，下一步应处理像 `af_056` 这种“哪一款最……”的全局比较问题。

方向建议：

- 对“最省电 / 最适合 / 哪款更值得买”这类 query，适当扩大最终保留候选；
- 避免 rerank 过早把跨型号比较所需的关键 doc 剪掉；
- 必要时对比较类 query 使用不同的 merge / rerank 策略。

## 结论摘要

1. `hybrid_rerank` 对当前小样本空气炸锅知识库是有价值的，应保留。例子：`af_001`、`af_021`、`af_043`。
2. “型号未确认却默认成某型号”是系统性问题，不是单点异常。例子：`af_001`、`af_011`、`af_019`。
3. 知识外问题目前总体是保守兜底，不是当前最严重的问题。例子：`af_053`、`af_055`、`af_057`、`af_058`。
4. `hybrid_rerank` 在部分全局比较类问题上存在回归，但优先级低于型号护栏。例子：`af_056`。
5. 因此，下一步实现优先级应为：
   1. 型号未确认场景修复
   2. 比较类 query 的 rerank 策略优化
