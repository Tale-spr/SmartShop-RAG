# 知识库与测试集扩充计划

## 背景

当前仓库的实验已经足以说明几个事实：

- 当前稳定基线仍然是 `hybrid_rerank`
- `weighted_rrf` 有探索价值，但在现有数据规模下容易被少量样本放大
- 继续在小知识库和小标注集上深调参数，收益已经明显下降

当前数据规模为：

- 知识库型号数：`5`
  - `MF-KZ30E201`
  - `MF-KZC6054`
  - `MF-KZE459X9BD`
  - `MF-KZE5004`
  - `MF-KZE7001`
- 品牌公共规则文档：`4`
  - `returns / warranty / shipping / invoice`
- 主测试集 `main_v2`：`45` 条
- 型号确认专项集 `model_confirmation_v1`：`19` 条
- Ragas 标注集：`18` 条

结论很明确：下一阶段的重点不应再是继续细调融合参数，而应先扩充 **知识库规模** 与 **测试覆盖度**，让后续检索实验结论更稳定、更可解释。

## 目标

本轮扩充的目标不是追求“大而全”，而是把当前项目从“小样本原型”推进到“可支撑检索策略比较”的规模。

目标分为两层：

### 1. 知识库目标

- 保持品类不变，继续聚焦 `空气炸锅`
- 保持品牌主线不变，优先继续扩 `美的`
- 将型号数从 `5` 个扩到 `8-12` 个
- 每个型号继续维持统一的 `cleaned` 文档结构：
  - `detail.md`
  - `specs.md`
  - `manual.md`
  - `service.md`
- 公共规则文档继续复用 `shared/policies`

### 2. 测试集目标

- 将 `main_v2` 从 `45` 条扩到 `70-90` 条
- 将 `model_confirmation_v1` 从 `19` 条扩到 `30-36` 条
- 将 Ragas 标注集从 `18` 条扩到 `30-36` 条

目标不是让所有集合无限增长，而是让每一类问题都有足够样本支撑趋势判断。

## 扩充原则

### 1. 先扩“可区分的型号差异”，不是先扩文档数量

优先补这些差异维度：

- 容量梯度更密
  - `3L / 4L / 4.5L / 5L / 5.5L / 6L / 6.5L / 7L`
- 操作方式差异
  - `双旋钮`
  - `触控/电子控制`
- 结构差异
  - `可视窗 / 非可视窗`
  - `方形烤篮 / 圆形烤篮`
- 核心功能差异
  - `双热源 / 普通款`
  - `自动断电 / 非自动断电`

原因：

- 这些差异会直接影响 `BM25` 对关键词的价值
- 也会直接影响 `vector / BM25 / RRF` 在弱特征 query 上的表现
- 它们比单纯多几页相似说明书更有实验价值

### 2. 先扩当前品牌，再考虑跨品牌

本轮不建议立刻加入第二品牌。

原因：

- 当前最需要解决的是“同品牌不同型号之间的区分与排序”
- 继续扩 `美的` 更利于验证：
  - 型号识别
  - 弱特征检索
  - BM25 的词面补召回价值
- 过早引入第二品牌会把问题复杂度从“型号区分”扩大到“品牌差异 + 规则差异 + 语料风格差异”

因此本轮默认策略是：

- **只扩美的，不扩第二品牌**

### 3. 测试集扩充优先于检索调参

在完成以下扩充前，不再继续深调：

- `vector_weight / bm25_weight`
- `rrf_k`
- 更复杂的 manual 偏置
- 更多 query bucket

原因：

- 当前这些调参已经开始受小样本波动影响
- 先把样本量拉起来，结论才更可靠

## 知识库扩充计划

### 第一阶段：补齐 3-5 个新型号

建议新增 `3-5` 个美的空气炸锅型号，优先选择现有知识库中尚未覆盖的“相邻差异型号”。

筛选标准：

- 与现有 `5` 个型号形成明确对照
- 官网或商品页仍能稳定拿到：
  - 详情
  - 规格参数
  - 服务支持
  - 手机版说明书
- 尽量避免只是同款换配色或轻微营销变体

优先补的方向：

- `4L` 左右的小容量款
- `6L / 6.5L` 的过渡容量款
- `电子控制但无双热源` 的中间型
- `有可视窗但非高端双热源` 的中段型

### 第二阶段：沿用现有采集与清洗流水线

新型号继续沿用当前目录结构：

```text
data/knowledge_base/air_fryer/midea/<MODEL>/
  cleaned/
    detail.md
    specs.md
    manual.md
    service.md
```

处理规则保持不变：

- `raw` 只用于采集过程，不作为运行知识库依赖
- 最终只 ingest `cleaned/` 文档
- `shared/policies` 仍作为公共规则层

### 第三阶段：知识文档统一检查

扩完型号后统一做一轮一致性检查，确保：

- 各型号 `doc_type` 命名完全一致
- 容量、功率、可视窗、操作方式等关键字段表达风格统一
- `manual.md` 中常见问题标题尽量保留原文表达
- `detail.md` 不混入价格、促销、优惠信息

这一步的目标是提高后续 chunk 和检索的一致性，避免新扩的数据把语料风格拉散。

## 测试集扩充计划

### 1. 主测试集 `main_v2`

将 `45` 条扩到 `70-90` 条。

新增样本优先覆盖：

- 新增型号对应的参数类问题
- 容量词和弱特征问题
  - `3L / 4L / 6L / 6.5L / 7L`
  - `可视窗`
  - `旋钮`
  - `双热源`
- 多型号比较问题
- manual 导向问题
  - 首次使用
  - 清洁保养
  - 常见故障

扩充后建议类别分布：

- 商品参数类：`12-15`
- 适用场景类：`10-12`
- 功能差异类：`12-15`
- 使用入门类：`8-10`
- 常见问题/故障类：`12-15`
- 多型号对比类：`8-10`
- 售后规则类：`4-6`

### 2. 型号确认专项集 `model_confirmation_v1`

将 `19` 条扩到 `30-36` 条。

重点增加三类样本：

- `unconfirmed`
  - 用户只给容量、颜色、可视窗、旋钮等弱特征
- `explicit_confirmed`
  - 用户明确给出型号后，系统应直接回答
- `conflicted`
  - 用户给出型号，但又提供与该型号冲突的特征描述

这一步的目标不是增加更多常规问答，而是让“型号确认能力”真正形成一个专项测试集。

### 3. Ragas 标注集

将 `18` 条扩到 `30-36` 条。

优先扩充以下类别：

- 适用场景类
- 常见问题/故障类
- 多型号对比类
- 售后规则类

标注策略：

- 继续采用最小字段：
  - `id`
  - `reference`
- 不把 reference 写成长段商品介绍
- reference 必须直击问题核心，便于 `AnswerRelevancy` 和 `Faithfulness` 判断

标注原则：

- 规则类：先给结论，再补条件
- 场景类：只写与当前判断相关的信息
- 故障类：不额外补机理解释
- 比较类：先给购买建议，再给依据

## 执行顺序

### 第一步：先扩测试集，再扩知识库

建议先补：

- `main_v2`
- `model_confirmation_v1`
- `main_v2_reference_answers_v1`

原因：

- 成本低
- 见效快
- 可以先让现有知识库的评测结论更稳

### 第二步：再扩 3-5 个型号

完成数据采集、清洗、统一检查后：

- 重建向量库
- 重建 BM25 索引
- 重新跑主实验

### 第三步：最后再回到检索策略实验

扩充完成后再重新比较：

- `hybrid_rerank`
- `weighted_rrf_rerank`
- `weighted_rrf_v2_rerank`

到那时再判断：

- weighted RRF 是否真的有持续价值
- 还是当前结论只是小样本现象

## 验收标准

本轮扩充完成后，至少满足：

- 美的空气炸锅型号数达到 `8-12`
- `main_v2` 达到 `70-90` 条
- `model_confirmation_v1` 达到 `30-36` 条
- Ragas 标注集达到 `30-36` 条
- 新增型号的 `cleaned` 文档结构统一
- 扩充后至少重新跑一次：
  - 主实验
  - Ragas 主评测

## 当前默认决策

本计划默认采用以下策略：

- 不新增第二品牌
- 不继续细调 RRF 参数
- 先扩测试集，再扩知识库
- 先补 `main_v2` 与 Ragas 标注，再补新型号

这是当前最务实、对后续实验可信度提升最大的路线。
