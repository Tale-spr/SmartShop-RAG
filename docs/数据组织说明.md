# 数据组织说明

## 知识库结构

项目当前围绕空气炸锅品类构建知识库，目录结构按品牌、型号和文档类型组织：

```text
data/knowledge_base/
└─air_fryer/
   └─midea/
      ├─MF-xxxx/
      │  ├─raw/
      │  └─cleaned/
      └─shared/
         └─policies/
```

其中：

- `raw/`：原始采集与整理过程文件
- `cleaned/`：进入检索主链路的结构化文本
- `shared/policies/`：跨型号共用的规则信息

## 文档类型

当前每个型号下的知识通常包含：

- `detail.md`：商品卖点、适用场景、功能说明
- `specs.md`：关键参数与规格
- `manual.md`：使用说明、清洁保养、常见问题
- `service.md`：售后、发货、发票、保修等信息

进入检索链路的主数据源是：

- 各型号 `cleaned/*.md`
- `shared/policies/*.md`

## 查询与测试集

项目中的问题集分为三层：

- `query_bank_v2`：通用题库
- `main_v3`：主检索测试集
- `model_confirmation_v2`：型号确认专项集
- `ragas_v2`：Ragas 标注候选集

这样做的目的是把“知识覆盖”“主链路评测”“业务专项测试”分开，避免一个题库同时承担多种职责。

## 向量库元数据

文档进入向量库后，会补充结构化元数据，例如：

- `brand`
- `model`
- `doc_type`
- `source_path`
- `chunk_index`
- `chunk_id`

这些字段用于：

- 检索 trace 解释
- 候选结果分析
- 后续的融合与过滤策略实验

## 为什么保留 raw 和 cleaned 两层

项目对外展示的主链路只使用 `cleaned/` 文档，但保留 `raw/` 有两个目的：

- 保留公开知识来源到结构化知识的整理路径
- 便于后续继续补充、修订和抽查原始信息

因此，`cleaned/` 代表可检索知识，`raw/` 代表可追踪来源。
